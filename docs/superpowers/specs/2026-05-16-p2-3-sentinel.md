# P2-3 Sentinel 限流调通 — 设计文档

> 版本：v1.0 | 日期：2026-05-16 | 状态：已确认，待实施

## 背景

Gateway 的 `SentinelRulesConfig` 已定义 4 条限流规则（login=10 / submit=5 / ai=3 / default=100 QPS），但从未验证是否实际生效。全文 0 处 `@SentinelResource` 使用，微服务内部无限流兜底。

## 目标

1. 3 个关键 Service 方法加 `@SentinelResource` + fallback 兜底
2. 1 个单元测试验证 Gateway Sentinel 规则加载 + BlockHandler 正确

## @SentinelResource 标注

| 服务 | 方法 | 资源名 | fallback 行为 |
|---|---|---|---|
| expense-service | `ExpenseReportServiceImpl.submit()` | `expense_report_submit` | 返回 "系统繁忙，请稍后重试" |
| approval-service | `ApprovalProcessServiceImpl.startProcess()` | `approval_process_start` | 返回 "审批服务繁忙，请稍后重试" |
| ai-service | `ReviewServiceImpl.review()` | `ai_review` | 返回保守审核结果（APPROVED, LOW） |

fallback 方法签名与原始方法一致，触发限流/异常时自动降级。

## 验证测试

新建 `SentinelRulesConfigTest`（gateway-service）：

| 测试 | 方式 |
|---|---|
| 规则加载验证 | `FlowRuleManager.getRules()` 断言 4 条规则 + 各 QPS 值 |
| BlockHandler 响应格式 | 构造被限流场景，断言返回 JSON `code:429` |

不走实际 HTTP 压测。
